## 一、产品定位与商业模式

### 1.1 核心定位

Token Plan 团队版（企业版）是阿里云百炼面向企业用户推出的 AI 模型订阅服务，与面向个人用户的个人版并行存在。其商业模式核心是：

- **预付制订阅** — 按月/按年购买坐席，绑定固定 Credits 配额
- **Credits 作为统一计量货币** — 屏蔽不同模型间的价格差异，企业只需关注 Credits 消耗
- **弹性补充机制** — 通过共享用量包应对突发流量，不强制升档
- **严格当月清零** — 三类额度均到期清零，无任何跨月结转

### 1.2 Credits 定价体系 ✅ 高可信

**数据来源**：官方帮助文档 + 官方开发者社区 + 多个第三方博客交叉印证，数字数据完全一致。

#### 坐席定价（月付）

| 坐席档次 | 月费 | Credits 配额/月 | 单价（元/Credit） | 适用人群参考 |
|---------|------|--------------|----------------|-----------|
| 标准坐席 | ¥198 | 25,000 | 0.00792 | 轻度使用，产品/测试类 |
| 高级坐席 | ¥698 | 100,000 | 0.00698 | 日常高频，工程师/分析师 |
| 尊享坐席 | ¥1,398 | 250,000 | 0.005592 | 重度依赖，AI工程师/平台开发 |

> **注意**：坐席名称在不同来源中存在差异（旗舰版/尊享版），官方正式名称为「标准坐席」「高级坐席」「尊享坐席」，数字数据无争议。

**档位价格梯度**：
- 高级坐席 vs 标准坐席：用量 4× 但价格仅 3.5×，性价比更优
- 尊享坐席 vs 标准坐席：用量 10× 但价格仅 7×，大量使用时优先升档而非买用量包

## 二、Credits 抵扣细粒度规则

### 2.1 Credits 消耗影响因素 ✅ 高可信

Credits 消耗量**并非固定值**，由以下四项因素动态决定：

| 因素 | 说明 |
|------|------|
| 模型类型 | 不同模型换算比例不同（具体完整比例表官方未公开） |
| Token 用量 | 输入 Token + 缓存 Token + 输出 Token 分别按各自比例折算 |
| 思考模式 | 开启 Extended Thinking 后消耗更多 Credits |
| 工具调用 | Function Call / 代码执行等工具调用会额外消耗 Credits |

### 2.2 缓存 Token vs 普通 Token 抵扣差异 ✅ 高可信

| Token 类型 | 换算比例 | 相对折扣 |
|-----------|---------|---------|
| 普通输入 Token | 约 5,000 Token = 1 Credit | 基准 1× |
| 缓存命中 Token | 约 50,000 Token = 1 Credit | **约 1/10 折扣** |
| 输出 Token | 约 714 Token = 1 Credit（输出比输入贵约 7×） | — |

**实测示例**（Qwen3.6-Plus 单次请求）：

```
普通输入  8,349 tokens  → 1.67 Credits
缓存命中 40,794 tokens  → 0.82 Credits  ← 缓存效率极高（约 1/10）
输出       573 tokens  → 0.69 Credits
合计                    → 3.18 Credits
```

> ⚠️ **信息盲区**：不同模型（Qwen3 系列各档位、DeepSeek、多模态模型）的完整 Credits 换算比例表官方未完整公开；图片、音频等多模态 Token 的折算规则同样缺乏可验证信息。

## 三、坐席管理机制

### 3.1 核心规则 ✅ 高可信

**数据来源**：官方帮助文档 `help.aliyun.com/zh/model-studio/token-plan-team` 及 FAQ

| 规则维度 | 具体内容 |
|---------|---------|
| 一人一席 | 每个成员同一时间只能持有一个席位，席位不可跨成员共享，每个订阅仅支持一个专属 API Key |
| API Key 管控 | 分配坐席时系统自动生成，成员无法自行生成，须联系管理员获取明文 |
| Key 格式 | Token Plan 专属 Key 前缀为 `sk-sp-`，与普通按量付费 Key（`sk-` 前缀）**不兼容、不可互换** |
| 使用场景限制 | **仅限兼容 AI 工具的交互式使用，禁止用于自动化脚本或应用后端** |
| 席位到期 | 席位到期后对应 API Key 失效，需重新订阅并分配席位才能恢复 |
| 弹性补购 | 新增席位与现有订阅**统一到期**，费用按剩余时长折算 |

**弹性补购示例**：订阅周期剩余 15 天时加购标准坐席，仅收取 15 天费用（¥198 × 15/30 = ¥99）。

### 3.2 API Key 生命周期详解 ✅ 高可信

#### 生成与获取

```
管理员分配坐席给成员
      ↓
系统自动生成专属 API Key（sk-sp-xxx 格式）
      ↓
明文仅此刻完整显示一次 ← 管理员须当场记录并人工传递给成员
      ↓
之后控制台只显示脱敏格式（sk-sp-****），管理员也无法再查看明文
```

> ⚠️ **无系统内通知机制**：官方文档未说明有邮件/站内信推送，Key 的传递依赖管理员人工操作。

#### 坐席转移流程 ✅ 高可信

坐席在成员间转移是**管理员主动操作的两步流程**，非系统自动感知：

```
① 管理员点「回收席位」→ 确认弹窗 → 成员 A 失去使用权，席位变为「未分配」
   （无需成员 A 确认，管理员单方面即可执行）
            ↓
② 管理员重新分配给成员 B → 系统自动为成员 B 生成新 API Key
   （原成员 A 的 Key 不会转移给成员 B）
```

> ⚠️ **回收后 Key 的失效时机**：官方文档仅说明成员「失去使用权」，**未明确是否立即失效或存在宽限期**。「立即失效无宽限期」的说法经验证未能通过（1/2票），建议直接联系阿里云技术支持确认。

#### 续订场景

| 续订方式 | API Key 是否变更 |
|---------|----------------|
| 正常到期续费 | **文档未明确说明**，是否自动恢复原 Key 存在空白 |
| 退款后重新购买 | ✅ **Key 和 Base URL 均会变更**，必须重新配置 |

#### Key 丢失 / 泄露处理

- 管理员在成员管理页面点击「重置」→ 原 Key **立即失效** → 生成新 Key（同样只显示一次）
- 管理员事后**无法查看**任何 Key 的历史明文，只能重置

### 3.3 权限层级

```
管理员（Admin）
  ├── 分配/回收坐席（单方面操作，无需成员确认）
  ├── 管理席位购买、升降档、到期续订
  ├── 重置成员 API Key（原 Key 立即失效）
  ├── 配置共享用量包
  └── 分配账户配额至多个业务空间（Workspace）

团队成员（Member）
  ├── 使用管理员传递的独立 API Key（sk-sp- 前缀）
  ├── 消耗本坐席月度 Credits + 共享用量包
  └── ❌ 无法自行生成或查看 API Key
      ❌ 无法查看或管理其他成员的用量
```

### 3.4 多业务空间（Workspace）隔离 ✅ 高可信

**数据来源**：官方帮助文档 `help.aliyun.com/zh/model-studio/use-workspace`

账户总配额可按比例分配给多个业务空间，实现多项目隔离管理：

```
账户总配额：1000 QPM
  ├── prod-workspace：600 QPM（生产环境）
  ├── test-workspace：200 QPM（测试环境）
  └── dev-workspace：100 QPM（开发环境）
```

**企业价值**：不同项目、不同团队的用量互不干扰，避免单一项目流量突增影响全局。

### 3.5 尚待明确的问题

- 坐席总数量是否有上限？大型企业（100 席以上）是否有专属折扣或框架协议？
- 是否支持对公转账、合同签订、发票开具等企业采购合规流程？
- 是否支持降档（如从高级坐席降至标准坐席）或中途退款？
- 正常到期续费后，原 API Key 是否自动恢复可用？
- 坐席回收后（未分配期间），被回收成员的 Key 失效时机？
- 坐席转移期间（已回收但未分配）共享用量包额度如何处理？
- 升降档时 API Key 是否会变化？（官方文档完全未覆盖）

## 四、配额抵扣规则

### 4.1 三级抵扣优先级 ✅ 高可信

```
第一优先：坐席月度额度
      ↓ 耗尽后（不自动转按量）
第二优先：共享用量包（多个用量包时，按最近到期日优先消耗）
      ↓ 全部耗尽后
服务直接暂停 ──→ 只能：补购共享用量包 OR 等待下一计费周期
```

> ❌ **已被验证为不准确**（0/3 票）：「额度耗尽后自动转按量付费」  
> ❌ **已被验证为不准确**（0/3 票）：「可手动开启额度耗尽暂停功能」（不存在此开关，Token Plan 默认即耗尽即停）

### 4.2 所有额度清零规则汇总 ✅ 高可信

| 额度类型 | 有效期 | 到期处理 | 是否结转 |
|---------|------|---------|---------|
| 坐席月度 Credits | 订阅月周期 | 自动清零 | ❌ 不结转 |
| 共享用量包 Credits | 购买后 1 个月 | 自动清零 | ❌ 不结转 |
| 节省计划（Savings Plan）月度承诺 | 月承诺周期 | 自动过期 | ❌ 不可累积 |
| Coding Plan 月度额度 | 订阅周期 | 自动清零 | ❌ 不结转 |

> ⚠️ **高风险点**：四类额度均当月清零，月末剩余即损失。需建立用量预测机制，避免浪费。

### 4.3 节省计划（Savings Plan）与 Token Plan 的关系 ⚠️ 中可信

两套体系**相互独立**，节省计划内部抵扣顺序为：

```
免费额度 → 资源包 → 特定模型节省计划 → AI 通用型节省计划 → 按量付费
（同类计划中，先消耗先到期的）
```

> ⚠️ Token Plan 与节省计划是否可叠加购买并同时生效，官方文档**未明确说明**，是当前最大的商业模式盲区之一。按量付费兜底是节省计划体系的特性，**不适用于** Token Plan。

## 五、限流体系详解

### 5.1 三层限流结构 ✅ 高可信

| 层级 | 触发条件 | 错误信息 |
|------|---------|---------|
| **RPM / TPM**（分钟级） | 每分钟请求数或 Token 数超过模型限额 | `Requests rate limit exceeded` / `Allocated quota exceeded` |
| **RPS / TPS**（秒级） | 约等于 RPM/60 和 TPM/60 的瞬时值 | — |
| **流量突增保护** | 阈值随服务状态**动态调整**，非固定值，即便未达分钟级上限也可触发 | `Request rate increased too quickly` |

> 🔴 **流量突增保护最难预测**：阈值动态变化，正常业务也可能触发，必须在代码层实现指数退避重试（Exponential Backoff）。

### 5.2 限流计算维度 ✅ 高可信

**按主账号维度合并计算**，无法对子 Key 设置独立限速：

```
主账号限流额度 = 所有 RAM 子账号 + 所有 Workspace + 所有 API Key 的调用量之和
```

> ⚠️ 团队内任一成员的高频调用会消耗所有人共享的限流额度，需在应用层自行实现按成员/业务线的限流。

### 5.3 不同模型版本限额对比 ✅ 高可信（北京区域 `qwen-plus` 示例）

| 模型版本 | RPM | TPM | 风险等级 |
|---------|-----|-----|---------|
| `qwen-plus`（**稳定版**） | **30,000** | **500 万** | 低 |
| `qwen-plus-2025-07-28`（快照版） | **60** | **100 万** | 🔴 极高 |
| **差距** | **500 倍** | **5 倍** | — |

> 🔴 **高风险陷阱**：使用快照版（指定具体日期的版本号）时 RPM 仅为稳定版的 1/500。**生产环境务必只用稳定版别名**，禁止硬编码快照版版本号。

### 5.4 Coding Plan Pro 限流结构 ✅ 高可信

Coding Plan 采用三层嵌套限流，与 Token Plan 不同：

| 时间窗口 | 最大请求数 |
|---------|---------|
| 每 5 小时 | 6,000 次 |
| 每周 | 45,000 次 |
| 每月 | 90,000 次（到期清零不结转） |

> 注：Coding Plan Lite 已于 2026 年 4 月停售。

## 六、企业版 vs 个人版完整对比

### 6.1 核心差异对比表

| 维度 | 个人版 | 企业版（团队版） |
|------|------|--------------|
| **用户规模** | 单一用户 | 多席位，支持团队协作 |
| **席位管理** | 无 | 管理员统一管控，一人一席 |
| **API Key** | 用户自行生成 | 管理员分配，成员无法自行生成 |
| **Credits 配额** | 个人档位限制 | 三档坐席 + 共享用量包弹性叠加 |
| **业务空间隔离** | ❌ 不支持 | ✅ 支持多 Workspace 按比例分配配额 |
| **企业数据隐私** | 标准保障 | 企业级数据隐私保障 |
| **成本归因** | 个人维度 | 可按坐席/Workspace 追踪成本 |
| **配额共享** | 无 | 共享用量包可跨坐席使用 |
| **适用场景** | 个人开发、学习、小型项目 | 团队协作、企业项目、多业务线管理 |

### 6.2 选择决策树

```
团队人数 ≤ 1 人，且无多项目隔离需求？
  └── 是 → 个人版（更灵活，无席位管控开销）
  └── 否 ↓

需要 API Key 统一管控 / 成本归因到个人？
  └── 是 → 企业版（坐席管理机制天然支持）
  └── 否 ↓

需要多项目/多部门配额隔离？
  └── 是 → 企业版（Workspace 机制）
  └── 否 → 视用量规模选择个人版或企业版
```

## 七、风险清单与应对建议

### 7.1 高风险点

| 风险 | 描述 | 应对措施 |
|------|------|---------|
| 🔴 服务突然中断 | 额度耗尽后零通知直接停服，无任何兜底 | 建立月度用量预警（建议 70% 时触发告警），预留 30% 缓冲 |
| 🔴 快照版限流陷阱 | 快照版 RPM 仅为稳定版 1/500，生产环境极易限流 | 生产环境**只用稳定版别名**，禁止硬编码快照版本号 |
| 🔴 弹性包到期浪费 | 1 个月到期清零，且单价高于尊享坐席 | 按实际用量购买，避免月末大量剩余；月均超档 120% 时优先升档 |

### 7.2 中风险点

| 风险 | 描述 | 应对措施 |
|------|------|---------|
| 🟡 限流合并计算 | 一个成员高频调用影响全团队限流额度 | 应用层为每个业务线实现独立限流，不依赖平台 |
| 🟡 流量突增保护不可预测 | 突增保护阈值动态变化，正常业务也可能触发 | 实现指数退避重试，捕获 `Request rate increased too quickly` |
| 🟡 API Key 失效 | 席位到期后 API Key 立即失效 | 提前 7 天续订，避免生产环境中断 |
| 🟡 Credits 换算不透明 | 不同模型换算比例官方未完整公开 | 上线前对核心模型进行 Credits 消耗压测，建立实测换算表 |

### 7.3 成本优化机会

| 优化方向 | 预期收益 |
|---------|---------|
| 最大化缓存命中率 | 缓存 Token 约为普通 Token 的 1/10 费用，系统 Prompt 固定时收益极大 |
| 升档优于买弹性包 | 弹性包单价（0.008）> 尊享坐席单价（0.005592），月均超档时升档更划算 |
| 错峰批量任务 | 月初配额充足时集中执行高耗量任务，避免月末不足 |
| Workspace 分层管理 | 生产/测试/开发分 Workspace，防止测试流量抢占生产配额 |
| Prompt 精简 | RAG 场景控制 Context 长度，可降低 30–50% 输入 Credits |

### 7.4 采购决策参考

| 月人均 AI 调用量（估算） | 推荐档位 | 理由 |
|----------------------|---------|------|
| < 2 万 Credits/人/月 | 标准坐席 ¥198 | 用不满高级坐席，升档浪费 |
| 2–8 万 Credits/人/月 | 标准坐席 × N | 按需购买，弹性包补充峰值 |
| 8–20 万 Credits/人/月 | 高级坐席 ¥698 | 性价比优于标准 × 4 |
| > 20 万 Credits/人/月 | 尊享坐席 ¥1,398 | 单价最低（0.005592元/Credit） |

6. **模型访问差异**：企业版与个人版在模型访问权限上是否存在差异？
7. **合并限流是否有例外**：高价值客户是否可购买独立限流包，实现子 Key 级别的独立限速？
8. **企业采购合规**：是否支持发票开具、合同签订、对公转账付款？
