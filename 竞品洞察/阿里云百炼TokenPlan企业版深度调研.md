## 一、产品定位与商业模式

### 1.1 核心定位

Token Plan 团队版（企业版）是阿里云百炼面向企业用户推出的 AI 模型订阅服务，与面向个人用户的个人版并行存在。其商业模式核心是：

- **预付制订阅** — 按月/按年购买坐席，绑定固定 Credits 配额
- **Credits 作为统一计量货币** — 屏蔽不同模型间的价格差异，企业只需关注 Credits 消耗
- **弹性补充机制** — 通过共享用量包应对突发流量，不强制升档
- **严格到期清零** — 坐席月额度和共享用量包均到期清零，无跨周期结转

## 二、Credits 抵扣细粒度规则

### 2.1 Credits 消耗影响因素 ✅ 高可信

Credits 消耗量**并非固定值**，由以下四项因素动态决定：

| 因素 | 说明 |
|------|------|
| 模型类型 | 不同模型换算比例不同（具体完整比例表官方未公开） |
| Token 用量 | 输入 Token + 缓存 Token + 输出 Token 分别按各自比例折算 |
| 思考模式 | 开启 Extended Thinking 后消耗更多 Credits |
| 工具调用 | 内置工具本身不额外收费，但工具调用产生并进入上下文的 Token 仍参与 Credits 计量；MCP 服务可能使用另一套普通百炼凭证与计费链路 |

### 2.2 缓存 Token vs 普通 Token 抵扣差异 ✅ 高可信

| Token 类型 | 换算比例 | 相对折扣 |
|-----------|---------|---------|
| 普通输入 Token | 约 5,000 Token = 1 Credit | 基准 1× |
| 缓存命中 Token | 约 50,000 Token = 1 Credit | **约 1/10 折扣** |
| 输出 Token | 约 830 Token = 1 Credit（输出比输入贵约 6×） | — |

**官方计算示例**（`qwen3.6-plus` 单次请求）：

```
普通输入  8,349 tokens  → 1.67 Credits
缓存命中 40,794 tokens  → 0.82 Credits  ← 缓存效率极高（约 1/10）
输出       573 tokens  → 0.69 Credits
合计                    → 3.18 Credits
```

> ⚠️ **信息盲区**：不同模型（Qwen3 系列各档位、DeepSeek、多模态模型）的完整 Credits 换算比例表官方未完整公开；图片、音频等多模态 Token 的折算规则同样缺乏可验证信息。

## 三、坐席管理机制

### 3.1 核心规则（逐项标注未知边界）

**数据来源**：[团队管理](https://help.aliyun.com/zh/model-studio/token-plan-team)及[团队版 FAQ](https://help.aliyun.com/zh/model-studio/token-plan-faq)

| 规则维度 | 具体内容 |
|---------|---------|
| 一人一席 | 每个成员同一时间只能持有一个席位；分配席位后系统为该成员生成一个专属 API Key，Key 不可跨成员共享；底层究竟绑定成员 ID、坐席 ID 还是订阅实例未公开 |
| API Key 管控 | 手工添加成员时由管理员分配坐席并生成 Key；SSO/钉钉成员是否可自助管理 Key，官方 FAQ 与团队管理页口径不完全一致，需控制台实测 |
| Key 格式 | Token Plan 专属 Key 前缀为 `sk-sp-`，与普通按量付费 Key（`sk-` 前缀）**不兼容、不可互换** |
| 使用场景限制 | **仅限兼容 AI 工具的交互式使用，禁止用于自动化脚本或应用后端** |
| 席位到期 | 席位到期后对应 API Key 失效，需重新订阅并分配席位才能恢复 |
| 弹性补购 | 新增席位与现有订阅**统一到期**，费用按剩余时长折算 |

**弹性补购示例**：订阅周期剩余 15 天时加购标准坐席，仅收取 15 天费用（¥198 × 15/30 = ¥99）。

### 3.2 API Key 生命周期详解

#### 生成与获取

```
管理员分配坐席给成员
      ↓
系统自动生成专属 API Key（sk-sp-xxx 格式）
      ↓
明文仅此刻完整显示一次 ← 管理员须当场记录并人工传递给成员
      ↓
之后控制台只显示脱敏格式（sk-sp-****），管理员也无法再查看明文
```

> ❓ **Key 交付方式存在两种路径**：手工添加的成员由管理员交付 Key；SSO/钉钉成员可登录管理平台，但其是否能自助生成或重置 Key，官方页面口径不完全一致。

#### 坐席转移流程（流程明确，撤权时延未知）

坐席在成员间转移是**管理员主动操作的两步流程**，非系统自动感知：

```
① 管理员点「回收席位」→ 确认弹窗 → 成员 A 失去使用权，席位变为「未分配」
   （无需成员 A 确认，管理员单方面即可执行）
            ↓
② 管理员重新分配给成员 B → 系统自动为成员 B 生成新 API Key
   （原成员 A 的 Key 不会转移给成员 B）
```

> ❓ **回收后 Key 的失效时机**：官方文档仅说明成员“失去使用权”，未明确旧 Key 的撤销时刻或传播延迟，建议通过控制台测试或阿里云工单确认。

#### 续订场景

| 续订方式 | API Key 是否变更 |
|---------|----------------|
| 正常到期续费 | **文档未明确说明**，是否自动恢复原 Key 存在空白 |
| 退款后重新购买 | ✅ **Key 和 Base URL 均会变更**，必须重新配置 |

#### Key 丢失 / 泄露处理

- 管理员在成员管理页面点击“重置”→ 原 Key **立即失效** → 生成新 Key（同样只显示一次）
- 管理员事后**无法查看**任何 Key 的历史明文，只能重置

### 3.3 权限层级

```
管理员（Admin）
  ├── 分配/回收坐席（单方面操作，无需成员确认）
  ├── 管理席位购买、升降档、到期续订
  ├── 重置成员 API Key（原 Key 立即失效）
  ├── 配置共享用量包
  └── 查看组织、成员和模型维度的用量

团队成员（Member）
  ├── 使用本成员独立 API Key（sk-sp- 前缀）
  ├── 消耗本坐席月度 Credits + 共享用量包
  └── ❌ 不可查看或管理其他成员的用量
```

> 手工添加成员时由管理员生成并交付 Key；SSO/钉钉成员能否完整自助管理 Key，官方页面口径不一致，不能统一写成“所有成员均无法自行管理”。

### 3.4 Workspace 不是已证实的 Token Plan 隔离能力

普通百炼支持业务空间（Workspace）与配额管理，但当前 Token Plan 官方文档只确认按**成员、坐席和模型**查看 Credits 用量，没有确认可以把 Token Plan 坐席额度或共享用量包分配到 Workspace。

更重要的是，Token Plan FAQ 明确：技术限流按主账号聚合，账号下 RAM 子账号、业务空间和 API Key 的调用量合并计算。因此，即使普通百炼侧存在 Workspace，也不能据此推出 Token Plan 能实现项目级 Credits 隔离或吞吐隔离。

> **结论**：将 Workspace 用于 Token Plan 配额隔离属于未经证实的架构假设，不应作为采购卖点。企业版当前可确认的原生归因粒度是成员和模型，原生额度隔离粒度是坐席。

### 3.5 尚待明确的问题

- 坐席总数量是否有上限？大型企业（100 席以上）是否有专属折扣或框架协议？
- 是否支持对公转账、合同签订、发票开具等企业采购合规流程？
- 是否支持降档（如从高级坐席降至标准坐席）或中途退款？
- 正常到期续费后，原 API Key 是否自动恢复可用？
- 坐席回收后（未分配期间），被回收成员的 Key 失效时机？
- 坐席转移期间（已回收但未分配）共享用量包额度如何处理？
- 升降档时 API Key 是否会变化？（官方文档完全未覆盖）

## 四、配额抵扣规则

### 4.1 两级额度抵扣与暂停结果 ✅ 官方明确

```
第一优先：坐席月度额度
      ↓ 耗尽后（不自动转按量）
第二优先：共享用量包（多个用量包时，按最近到期日优先消耗）
      ↓ 全部耗尽后
服务直接暂停 ──→ 只能：补购共享用量包 OR 等待下一计费周期
```

> 官方规则不支持“额度耗尽后自动转按量付费”；服务暂停是抵扣失败后的结果，不是第三级额度。

### 4.2 Token Plan 企业版额度到期规则 ✅ 官方明确

| 额度类型 | 有效期 | 到期处理 | 是否结转 |
|---------|------|---------|---------|
| 坐席月度 Credits | 订阅月周期 | 自动清零 | ❌ 不结转 |
| 共享用量包 Credits | 购买后 1 个月 | 自动清零 | ❌ 不结转 |

> ⚠️ **高风险点**：坐席额度与共享用量包都不结转，但二者周期起点不同：坐席按订阅月，共享包从各自购买日计算一个月。需分别按到期时间告警，不能简单按自然月末管理。

### 4.3 节省计划（Savings Plan）与 Token Plan 的关系 ⚠️ 中可信

两套体系**相互独立**，节省计划内部抵扣顺序为：

```
免费额度 → 资源包 → 特定模型节省计划 → AI 通用型节省计划 → 按量付费
（同类计划中，先消耗先到期的）
```

> ⚠️ Token Plan 与节省计划是否可叠加购买并同时生效，官方文档**未明确说明**，是当前最大的商业模式盲区之一。按量付费兜底是节省计划体系的特性，**不适用于** Token Plan。

## 五、限流体系详解

### 5.1 可观察的三类限流

| 层级 | 触发条件 | 错误信息 |
|------|---------|---------|
| **RPM / TPM**（分钟级） | 每分钟请求数或 Token 数超过模型限额 | `Requests rate limit exceeded` / `Allocated quota exceeded` |
| **RPS / TPS**（秒级） | 约等于 RPM/60 和 TPM/60 的瞬时值 | — |
| **流量突增保护** | 阈值随服务状态**动态调整**，非固定值，即便未达分钟级上限也可触发 | `Request rate increased too quickly` |

> 🔴 **流量突增保护最难预测**：阈值动态变化，正常交互也可能触发。应优先使用兼容工具自带的退避与并发控制；若需要自建服务端重试和排队，应改用允许应用后端调用的普通百炼 API。

### 5.2 限流计算维度 ✅ 高可信

**按主账号维度合并计算**，无法对子 Key 设置独立限速：

```
主账号限流额度 = 所有 RAM 子账号 + 所有 Workspace + 所有 API Key 的调用量之和
```

> ⚠️ 团队内任一成员的高频调用会消耗共享限流额度。交互工具场景应限制成员并发；若需要业务线级服务端限流，应改用普通百炼 API 并在应用网关实现。

### 5.3 普通百炼模型版本限额示例（不可直接等同于 Token Plan）

| 模型版本 | RPM | TPM | 风险等级 |
|---------|-----|-----|---------|
| `qwen-plus`（**稳定版**） | **30,000** | **500 万** | 低 |
| `qwen-plus-2025-07-28`（快照版） | **60** | **100 万** | 🔴 极高 |
| **差距** | **500 倍** | **5 倍** | — |

> 该数值来自普通百炼北京区域限流文档，能说明“模型别名与快照版本的限流可能差异很大”，但官方没有确认 Token Plan 专属入口完整复用同一组数值。Token Plan 选型应使用套餐精确模型白名单，并通过允许的交互式工具实测，不能把本表当作套餐 SLA。

### 5.4 相邻产品旁注：Coding Plan Pro（不属于 Token Plan 抵扣链）

Coding Plan 采用三层嵌套限流，与 Token Plan 不同：

| 时间窗口 | 最大请求数 |
|---------|---------|
| 每 5 小时 | 6,000 次 |
| 每周 | 45,000 次 |
| 每月 | 90,000 次（到期清零不结转） |

> 注：Coding Plan Lite 已于 2026 年 4 月停售。

## 六、个人版 vs 企业版：差异不只在价格，而在整条调用链路

> **术语边界**：本章比较的是 2026 年上线的 **Token Plan 个人版**与 **Token Plan 团队版**。Coding Plan 是另一款按调用次数计量、使用 `coding.dashscope.aliyuncs.com` 端点的产品，不能把 Coding Plan 规则套到 Token Plan 个人版。

### 6.1 核心差异总表

| 维度 | Token Plan 个人版 | Token Plan 企业版（团队版） | 架构影响 |
|------|------------------|----------------------------|----------|
| 订阅主体 | 单个实名认证主体，同一主体限购一份 | 一个阿里云账号订阅，包含多个坐席 | 企业版多出组织、成员和坐席控制面 |
| 套餐档位 | Lite / Standard / Pro | 标准 / 高级 / 尊享坐席 | 两边档位和额度不可直接映射 |
| API Key 主体 | 个人订阅人专享，禁止共享 | 每个已分配坐席对应一个成员和一个专属 Key | 企业版可按成员撤权与归因 |
| Key 前缀 | `sk-sp-` | `sk-sp-` | 前缀相同不代表可互换，仍需匹配具体订阅 |
| 文本调用入口 | Token Plan OpenAI / Anthropic 兼容端点 | 同一组公开 Token Plan OpenAI / Anthropic 兼容端点 | **没有证据表明企业版拥有独立物理网关** |
| 核心额度 | 5 小时与 7 天两层 Credits 滑动窗口 | 每坐席每订阅月固定 Credits | 个人版是时间窗口控制，企业版是月度预算账本 |
| 超额补充 | 个人用量包；官方称超限后可购买继续使用 | 共享用量包，跨坐席使用 | 企业版需要坐席账本向共享账本回退 |
| 额度恢复 | 超过窗口时间后相应消耗自动释放；也可升级，个人用量包细则尚未公开 | 到下个订阅月重置；续费不补当前周期 | 恢复事件和告警策略完全不同 |
| 成员与角色 | 无组织成员体系 | 所有者、管理员、成员；支持手工、SSO、钉钉加入 | 企业版有独立控制面和权限审计要求 |
| 用量分析 | 个人套餐用量 | 组织趋势、模型用量、成员用量 | 企业版可按成员归因，但未证实支持 Workspace Credits 隔离 |
| 退订 | 暂不支持退订 | 未发生用量的坐席可按席位退订 | 资源回收流程不同 |
| 数据使用 | 输入与生成内容可用于服务改进和模型优化 | 承诺不使用对话数据训练模型 | 企业采购与合规决策的关键分界；见个人版概述与团队版 FAQ |
| 使用边界 | 仅限兼容工具中的交互式使用 | 同样仅限兼容工具中的交互式使用 | 两者都不应作为自动化脚本或应用后端 API 套餐 |

个人版当前档位进一步体现了“窗口制”差异：

| 个人版档位 | 当前限时月价 | 5 小时窗口 | 7 天窗口 | 官方建议并发 Agent |
|------------|--------------|------------|----------|--------------------|
| Lite | ¥39 | 700 Credits | 2,500 Credits | 1–2 个 |
| Standard | ¥139 | 3,000 Credits | 10,000 Credits | 3–4 个 |
| Pro | ¥499 | 12,000 Credits | 40,000 Credits | 6–8 个 |

任一窗口触顶都会暂停；超过 5 小时或 7 天的历史消耗自动释放。上述价格属于限时活动价格，采购时需重新核对。[个人版官方概述](https://help.aliyun.com/zh/model-studio/token-plan-personal-overview)

### 6.2 公开端点相同，但鉴权后的主体与账本不同

当前官方公开的 Token Plan 文本端点如下；个人版和团队版文档均指向这一组入口。[Base URL 官方总览](https://help.aliyun.com/zh/model-studio/base-url)

| 协议 | Base URL |
|------|----------|
| OpenAI 兼容 | `https://token-plan.cn-beijing.maas.aliyuncs.com/compatible-mode/v1` |
| Anthropic 兼容 | `https://token-plan.cn-beijing.maas.aliyuncs.com/apps/anthropic` |

个人版与企业版均属于 Token Plan 产品族，Key 都以 `sk-sp-` 开头。由此能确认的是**套餐凭证与普通百炼按量凭证存在逻辑隔离**：普通百炼 `sk-` Key、Coding Plan Key/Base URL、Token Plan Key/Base URL 不能混用，错配通常返回 401。

但以下说法不能从公开资料推出：

- 企业版运行在独立物理网关或独立集群；
- `sk-sp-` 字符串自身包含企业、成员或坐席 ID；
- 个人版和企业版一定经过两套不同的鉴权微服务；
- 每个成员 Key 拥有独立的 RPM/TPM 限流桶。

更稳妥的外部行为模型是：

```text
Token Plan 客户端
  │  sk-sp-* + Token Plan Base URL
  ▼
Token Plan 专属入口（官方可确认）
  │
  ├─ 个人版：校验个人订阅状态 → 检查 5 小时/7 天 Credits 窗口
  │
  └─ 企业版：校验订阅/成员/坐席状态 → 坐席 Credits → 企业共享用量包
       │
       └─ 所有成员仍受主账号聚合技术限流
```

其中“校验个人订阅”“校验成员/坐席”是依据外部状态行为作出的逻辑推导；阿里云没有公开内部服务拓扑、Token 载荷或鉴权缓存实现。

### 6.3 控制面与调用面的差异

#### 个人版

```text
个人购买套餐
  → 获得个人专属 Key
  → 在兼容工具配置 Key + Base URL
  → 调用时累计 5 小时与 7 天窗口 Credits
  → 任一窗口触顶后暂停
  → 等待滑动释放 / 升级
  → 或购买个人用量包（仅确认存在，抵扣链细节未公开）
```

个人版的核心对象可简化为“实名认证主体 → 个人订阅 → 专属 Key”。官方未公开 Key 精确绑定的是账号还是订阅实例，因此不应在内部设计文档中虚构更细的绑定字段。

#### 企业版

```text
阿里云账号购买团队订阅和坐席
  → 创建或接入组织成员
  → 给成员分配坐席
  → 系统生成成员专属 Key
  → 调用先扣成员坐席 Credits
  → 坐席额度不足后扣企业共享用量包
  → 管理员按成员与模型查看用量
```

企业版的外部对象关系可以明确为：

```text
阿里云账号
  └─ Token Plan 团队订阅
      └─ 企业组织
          ├─ 所有者 / 管理员
          └─ 成员
              └─ 一个已分配坐席
                  └─ 一个专属 API Key
```

### 6.4 配额与限流是两条不同的检查链

#### 个人版经济额度

```text
检查 5 小时 Credits 窗口
  → 检查 7 天 Credits 窗口
  → 任一触顶即暂停
  → 时间滑动后释放或升级套餐
  → 个人用量包可用于继续使用，但与双窗口的抵扣关系未公开
```

#### 企业版经济额度

```text
坐席月度 Credits
  → 用尽后检查共享用量包
  → 多个共享包优先扣最近到期者
  → 全部用尽后暂停，无按量付费兜底
```

#### 企业版技术限流

企业 FAQ 明确，技术限流按主账号聚合计算，主账号下 RAM 子账号、业务空间和 API Key 的调用量合并。[团队版 FAQ](https://help.aliyun.com/zh/model-studio/token-plan-faq) 由此产生一个重要结论：

> **独立成员 Key + 独立坐席 Credits，不等于独立吞吐能力。** 某个成员的突发调用仍可能挤占企业账号的共享 RPM/TPM/TPS 配额，影响其他成员。

此外，`429 AllocationQuota` / `insufficient_quota` 可能表示经济额度耗尽，也可能表示技术限流。调用工具不能仅凭 HTTP 状态码判断根因，必须结合控制台剩余额度、错误正文和限流指标。

### 6.5 API Key 与订阅对象生命周期矩阵

| 事件 | 个人版 | 企业版 | 对接入架构的影响 |
|------|--------|--------|------------------|
| 正常续费 | 续费仅延长订阅，不补当前窗口；是否更换 Key 未明确 | 到期前续费可避免服务中断，不补当前月额度；是否保留原 Key 未明确 | 不应自动假设续费必然换 Key或必然不换 |
| 到期 | 订阅失效后 Key 不可继续正常调用 | 坐席到期后对应 Key 无法调用 | 需把订阅有效性与 Key 健康状态分开监控 |
| 到期后重新购买 | 官方明确 Key 会变化，需重新配置 | 纯“到期后重购”的 Key 行为未单独说明 | 不要把到期重购与退订重购混为一谈 |
| 退款/退订后重购 | 个人版暂不支持退订 | FAQ 称 Key 和 Base URL 会变化，但公开固定端点与“变化”的对象/粒度存在解释缺口 | 明确会换 Key；Base URL 以重购后控制台接入信息为准，不假设公开端点必变 |
| 手动重置 Key | 当前个人版公开文档未说明是否支持及其失效规则 | 旧 Key 明确立即失效，新 Key 需重新分发 | 两版不能共用同一轮换假设 |
| 回收坐席 | 不适用 | 成员失去席位使用权；旧 Key 精确失效时点未公开 | 不要把“失去使用权”扩写成“零延迟撤销” |
| 重分配给新成员 | 不适用 | 系统生成新 Key，旧 Key 不会转交 | Seat ID 与 Credential ID 不应设计成同一个对象 |
| 移出组织 | 不适用 | 席位自动回收，Key 明确立即失效 | 属于成员级局部故障域 |
| 修改角色 | 不适用 | 不改变席位和组织归属；是否触发鉴权刷新未公开 | 角色权限与模型调用资格应分层建模 |
| 升级套餐/坐席 | 窗口上限立即提升；Key 是否变化未明确 | 按剩余时长补差价；Key、额度生效细节未完全公开 | 需要控制台实测后才能设计无感升级流程 |
| 额度耗尽 | Key 本身未必失效，但调用因窗口额度被阻断 | Key 本身未必失效，但坐席与共享包耗尽后阻断 | “凭证有效”和“可成功调用”是两个状态 |

> 团队版“先坐席、后共享包”是官方明确的账务优先级；单次请求跨越坐席剩余额度时是否在两个账本间原子拆扣，官方未说明。

### 6.6 个人版和企业版的共享故障域

| 故障或变更 | 影响范围 |
|------------|----------|
| 个人版到期或到期后重购换 Key | 该个人在所有已配置工具中的调用 |
| 企业成员 Key 重置/移出组织 | 单个成员及其所有工具 |
| 企业坐席额度耗尽但共享包可用 | 该成员继续从企业共享包消费，并开始影响全团队预算 |
| 企业共享包耗尽 | 所有已耗尽坐席无法继续调用 |
| 企业主账号触发聚合限流 | 可能影响账号下多个成员、Key和业务空间 |
| 企业订阅到期 | 多个坐席和成员形成企业级共同故障 |
| 企业退订重购 | Key 明确变化；Base URL 所指对象与变化粒度未解释，应按控制台新接入信息复核 |

### 6.7 对自建接入架构的直接影响

#### 凭证必须以组合配置管理

不能只保存 API Key 字符串，至少应把以下字段作为一个版本化配置单元：

```yaml
provider: aliyun-model-studio
plan_type: token-plan-personal | token-plan-team
protocol: openai | anthropic
base_url: https://...
api_key_secret_ref: secret://...
organization_id: optional
member_id: optional
seat_id: optional
status: active | quota_exhausted | expired | revoked | unknown
```

其中 `organization_id`、`member_id`、`seat_id` 是调用方自己的业务映射，不代表这些字段编码在阿里云 Key 中。

#### Key轮换必须校验完整配置组合

推荐流程：

```text
录入新 Key，并读取控制台当前 Base URL
  → 在兼容工具中做最小交互验证
  → 原子切换整组配置
  → 观察 401 / 429 / 模型不存在错误
  → 停止使用旧配置并保留审计记录
```

由于套餐禁止用于自动化脚本和应用后端，这里的“验证”和“切换”仅针对官方允许的交互式 AI 工具部署，不应演变成服务端流量代理。公开端点看似固定时也应复核，但不能反向断言每次 Key 轮换都会改变 Base URL。

#### 错误处理至少分五类

| 类别 | 典型原因 | 处置 |
|------|----------|------|
| 凭证/端点错误 | Key与Base URL不属于同一方案、Key复制错误 | 校验 `plan_type + protocol + base_url + key` 组合 |
| 生命周期撤权 | 到期、重置、移出组织、席位状态变化 | 查订阅、成员、坐席和Key状态 |
| 经济额度不足 | 个人窗口触顶；企业坐席与共享包耗尽 | 等待窗口释放、升级或补购适用额度 |
| 技术限流 | 主账号聚合 RPM/TPM/TPS 或突增保护 | 降低并发、平滑请求；不能靠买Credits解决 |
| 模型/协议错误 | 模型不在白名单、OpenAI与Anthropic路径错配 | 按套餐白名单与协议配置修正 |

#### 企业版需要双层观测

```text
成员层：成员 → 坐席 → Key → 模型 → Credits
企业层：订阅 → 共享用量包 → 主账号聚合限流
```

只看成员 Credits 无法解释主账号限流，只看企业总量又无法定位哪个成员触发共享包消耗。

### 6.8 选型结论

- 单人使用、接受数据用于服务改进、希望用滑动窗口控制成本：选择个人版。
- 需要组织成员、坐席撤权、成员用量归因以及“不用于训练”的数据承诺：选择企业版。
- 需要应用后端、自动化脚本、批处理或稳定服务 SLA：**两者都不适合**，应使用普通百炼按量 API、节省计划或专用部署。[团队版使用限制](https://help.aliyun.com/zh/model-studio/token-plan-faq)
- 需要项目级硬配额或每个 Key 独立 RPM/TPM：企业版现有公开能力仍不满足；若是后端业务，应改用普通百炼 API 并在应用网关实现，交互工具场景则需另行确认企业定制能力。
- 不能因为企业版提供成员独立 Key，就推断其不存在共享故障域；主账号聚合限流和企业订阅有效期仍会影响多个成员。

## 七、风险清单与应对建议

### 7.1 高风险点

| 风险 | 描述 | 应对措施 |
|------|------|---------|
| 🔴 服务突然中断 | 额度耗尽后服务暂停且无按量兜底；公开文档未说明平台告警策略 | 自建月度用量预警（建议 70% 时触发告警），预留 30% 缓冲 |
| 🔴 快照版限流陷阱 | 普通百炼文档显示部分快照版限额显著低于稳定版，但是否完整适用于 Token Plan 专属入口需复核 | 在官方允许的交互工具中优先使用套餐白名单内稳定版，并以 Token Plan 实测限流为准 |
| 🔴 弹性包到期浪费 | 1 个月到期清零，且单价高于尊享坐席 | 按实际用量购买，避免月末大量剩余；月均超档 120% 时优先升档 |

### 7.2 中风险点

| 风险 | 描述 | 应对措施 |
|------|------|---------|
| 🟡 限流合并计算 | 一个成员高频调用影响全团队限流额度 | 交互工具侧限制成员并发；后端业务改用普通百炼 API 后再做业务线限流 |
| 🟡 流量突增保护不可预测 | 突增保护阈值动态变化，正常交互也可能触发 | 使用兼容工具的退避能力；需要自建重试队列时改用普通百炼 API |
| 🟡 API Key 失效 | 席位到期后 API Key 无法调用；精确失效时刻和传播延迟未公开 | 提前续订，避免交互工具集中中断 |
| 🟡 Credits 换算不透明 | 不同模型换算比例官方未完整公开 | 上线前对核心模型进行 Credits 消耗压测，建立实测换算表 |

### 7.3 成本优化机会

| 优化方向 | 预期收益 |
|---------|---------|
| 最大化缓存命中率 | 缓存 Token 约为普通 Token 的 1/10 费用，系统 Prompt 固定时收益极大 |
| 升档优于长期买弹性包 | 弹性包单价（0.008）高于当前三个坐席档位，稳定超额时升档通常更划算 |
| 分散高耗量交互任务 | 避免少数成员短时耗尽坐席额度并大量占用共享包；不得用于违规批处理 |
| 成员级用量治理 | 按成员查看消耗并设置组织内部预算告警；Workspace 并非已证实的 Token Plan Credits 隔离能力 |
| Prompt 精简 | RAG 场景控制 Context 长度，可降低 30–50% 输入 Credits |

### 7.4 采购决策参考

| 月人均 AI 调用量（估算） | 推荐档位 | 理由 |
|----------------------|---------|------|
| < 2 万 Credits/人/月 | 标准坐席（原价 ¥198，当前限时 ¥150） | 用不满高级坐席，升档浪费 |
| 2–8 万 Credits/人/月 | 高级坐席或标准坐席＋短期共享包 | 同一成员不能叠加多个坐席，需比较升档与临时补包 |
| 8–20 万 Credits/人/月 | 高级坐席（原价 ¥698，当前限时 ¥550） | 当前限时单位 Credits 成本最低 |
| > 20 万 Credits/人/月 | 尊享坐席 ¥1,398 | 单价最低（0.005592元/Credit） |

## 八、尚待厘清的开放问题

1. **完整 Credits 换算比例表**：Qwen3 系列各档位、DeepSeek 系列、多模态模型（图片/音频）如何折算为 Credits？
2. **个人用量包细则**：价格、Credits、有效期、抵扣顺序，以及它与 5 小时/7 天窗口如何交互？
3. **正常续费后的 Key**：个人版和团队版是否都保持原 Key；团队席位是否需要重新分配？
4. **回收坐席的撤权传播**：回收后旧 Key 在鉴权面何时失效，是否存在缓存或宽限期？
5. **升级时的凭证行为**：个人版升级、企业坐席升级是否保持原 Key，额度从哪个时刻生效？
6. **退订重购时 Base URL 变化粒度**：企业 FAQ 称 Base URL 会变化，但公开文档给出统一端点，两者如何解释？
7. **跨账本原子抵扣**：单次请求超过坐席剩余 Credits 时，能否在同一请求内拆分扣坐席额度和共享包？
8. **失败请求计量**：流式中断、工具调用失败、模型侧错误时如何结算 Credits？
9. **成员自助 Key 权限**：SSO/钉钉成员是否可以生成或重置 Key，FAQ 与团队管理页口径需控制台验证。
10. **吞吐定制能力**：是否可为企业购买独立限流或为成员 Key 设置硬 RPM/TPM 上限？
11. **Token Plan 与节省计划关系**：两者能否在同一账号同时持有，以及是否完全走独立请求入口和账本？
12. **弹性包与坐席规模限制**：共享包能否多次购买突破单次 1,000 个上限，大型团队是否存在坐席上限或框架价？

## 九、已纠正的常见误解

| 误解 | 官方资料支持的正确表述 |
|------|------------------------|
| 额度耗尽后自动切换按量计费 | Token Plan 套餐额度耗尽后暂停，无按量兜底 |
| 企业版每个订阅只有一个 Key | 每个已分配坐席绑定一个成员并对应一个专属 Key |
| 成员独立 Key 等于成员独立限流 | 企业 FAQ 明确技术限流按主账号聚合，成员之间仍可能互相影响 |
| 企业版有原生 Workspace Credits 隔离 | 当前仅确认成员/模型用量分析和坐席额度；Workspace Credits 隔离未获 Token Plan 文档支持 |
| 企业版使用独立于个人版的公开 Base URL | 当前个人版与团队版公开文档给出相同 Token Plan 文本端点；内部实现是否分离未知 |
| 个人版就是 Coding Plan | 两者是独立产品：Token Plan 个人版按 Credits 双滑窗，Coding Plan 按调用次数且使用 coding 专属端点 |
| 个人版支持手动额度重置 | 当前公开文档只确认窗口自动释放、升级和个人用量包，未说明手动重置功能 |
| 内置工具必然额外收取固定 Credits | 内置工具本身不额外收费，相关 Token 进入统一 Credits 计量；MCP 可能走普通百炼凭证 |
| 席位回收后旧 Key 已确认零延迟失效 | 官方只确认成员失去使用权；“移出组织”和“重置 Key”才明确写了立即失效 |
| 普通百炼快照版限流表就是 Token Plan SLA | 普通百炼数值只能作为风险提示，Token Plan 专属入口的完整适用性未公开 |
