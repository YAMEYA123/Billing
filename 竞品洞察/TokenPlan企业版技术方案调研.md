## 一、行业背景：计费范式的结构性转变

AI 服务计费正从**按人头 SaaS 订阅**向**按 Token 消耗（TaaS）**发生结构性转变。

**核心数据（✅ 高可信）**

- Gartner 预测：2027 年 70% 的 SaaS 厂商将提供消费型定价
- 中国市场：2026 年 3 月日均 Token 消耗量达 **140 万亿**，较 2024 年初（1000 亿）增长超 1000 倍
- 三大运营商已推出整合百度、阿里、字节、DeepSeek 模型的 Token 订阅套餐
- 中国本土 AI 平台已由按次计费全面转向按 Token 计费

**驱动因素**：工作负载升级为 Agent 协作、代码生成等高耗 Token 场景，简单订阅模式无法准确反映资源消耗。

## 二、主流平台 TokenPlan 方案横向对比

### 2.1 阿里云百炼 Token Plan ✅ 高可信

**数据来源**：阿里云帮助中心官方文档 + 多个中文社区交叉印证

#### 坐席定价

| 档次 | 月费 | Credits 配额 | 折算单价 |
|------|------|------------|---------|
| 标准版 | ¥198/月 | 25,000 Credits | ¥7.92/千 Credits |
| 专业版 | ¥698/月 | 100,000 Credits | ¥6.98/千 Credits |
| 旗舰版 | ¥1,398/月 | 250,000 Credits | ¥5.59/千 Credits |

#### 共享弹性包

- 定价：¥5,000/个，625,000 Credits
- 有效期：**一个月，到期自动清零**（⚠️ 高风险点）

#### 抵扣顺序

```
月度坐席配额 → 共享弹性包（优先消耗最先到期者）→ 耗尽后服务暂停
```

#### 实际计费示例（Qwen3.6-Plus）

| 操作类型 | Token 数 | Credits 消耗 |
|---------|---------|------------|
| 输入 | 8,349 tokens | 1.67 Credits |
| 缓存命中 | 40,794 tokens | 0.82 Credits（大幅优惠） |
| 输出 | 573 tokens | 0.69 Credits |

**机制特点**

- Credits 作为统一抵扣单位，屏蔽不同模型间的价格差异
- 坐席配额独立，防止单一用户耗尽团队额度
- 共享弹性包可跨坐席消耗，应对突发流量

### 2.2 Azure OpenAI ✅ 高可信

**数据来源**：微软官方多租户架构文档 + Azure AI 生产指南 + PTU 计费文档

#### 核心配额机制

- 计量单位：**TPM（每分钟 Token 数）**，按 Deployment 分配
- 两种计费模式：

| 模式 | 特点 | 适用场景 |
|------|------|---------|
| 按需 TPM | 弹性付费，有速率限制 | 波动性负载 |
| PTU（预留吞吐量单元） | 按小时收费，**无论是否消耗均计费** | 稳定基线负载 |

#### 企业最佳实践：PTU + 按需 TPM 混合架构

```
PTU Deployment ──── 处理预期基线负载（成本可控）
     ↓ 溢出时
按需 TPM Deployment ─ 承接流量峰值（按实际消耗付费）
```

#### 多租户隔离方案

- 为每个租户创建独立 Deployment → 实现 TPM 配额隔离 + 精确成本归因
- PTU 可分配给特定客户，实现资源预留隔离

> ⚠️ 注意：「Azure OpenAI 平台层无内置租户级计费隔离，必须在应用层实现」这一说法**已被验证为不准确**。独立 Deployment 本身即可实现原生租户级隔离。

## 三、核心技术挑战

### 3.1 Token 成本悖论 ✅ 高可信

**现象**：单位 Token 价格持续下降，但企业 AI 运营支出反而上升。

**根因**：工作负载升级，Token 消耗量级跨越式增长：

| 场景类型 | 相对 Token 消耗量 | 数据来源 |
|---------|----------------|---------|
| 单轮问答 | 基准 1× | — |
| 多轮对话聊天 | ~1,200× | arXiv 2604.22750 |
| 编码类 Agent 任务 | ~3,500× | 斯坦福数字经济实验室 |
| 企业 Agent 工作流 | 50–500× | EY 研究 |

**核心公式**：
```
实际支出 = 单位Token价格（↓） × 实际消耗量（↑↑↑）
```
价格降幅远小于用量增幅，净支出持续上升。

### 3.2 Credits 抵扣机制优缺点分析

| 维度 | 优点 | 缺点 / 风险 |
|------|------|-----------|
| 预算可控性 | 月度上限明确，无意外账单 | 弹性包到期清零，突发业务易造成浪费 |
| 多模型统一 | Credits 屏蔽不同模型定价差异 | 跨模型换算规则不透明，难以直接对比 |
| 抵扣顺序 | 自动优先消耗将过期的包 | 复杂业务场景难以准确预测耗尽时点 |
| 坐席隔离 | 每坐席独立配额，防止一人耗尽全团队 | 共享包无法细粒度归因到具体个人 |

### 3.3 多租户成本归因难题

在企业多部门/多客户场景下，需要在**应用层**主动埋点实现：
- 按部门 / 项目 / 客户维度的 Token 用量追踪
- 成本分摊与报表输出
- 配额告警与限流

**Azure OpenAI** 通过独立 Deployment 在平台层解决了这一问题；**阿里云百炼** 坐席机制提供了一定程度的隔离，但共享弹性包的精细归因仍需应用层配合。

## 四、安全风险：AI 中转平台不可用于企业级

⚠️ **中可信**（调查性报道 + 学术研究支撑，但无精确量化数据）

**风险来源**：新浪财经/证券时报 2026 年 5 月调查报道 + 牛津中国政策实验室研究

| 风险类型 | 具体表现 |
|---------|---------|
| 数据安全 | 部分平台真实商业模式是出售用户提示词日志，低价是获客手段 |
| 服务稳定性 | 封号跑路、突然停服风险 |
| 质量欺诈 | 高峰期动态路由到低端模型（❌ 此点具体机制未通过验证） |
| 合规风险 | 上海警方 2026 年 5 月已拘留一名 Token 中转平台运营者 |

**结论**：AI 中转平台绝对不应进入企业级选型范围，无论价格多具吸引力。

## 五、成本优化最佳实践

### 5.1 可立即实施

| 措施 | 预期效果 | 适用平台 |
|------|---------|---------|
| 启用 Prompt 缓存 | 降低 50–90% 输入成本 | Anthropic（90%折扣）、OpenAI（50%折扣） |
| DeepSeek 错峰调度 | 批量任务节省 50% | DeepSeek |
| PTU + 按需 TPM 混合 | 消除 PTU 低谷期浪费 | Azure OpenAI |

### 5.2 架构层面

| 措施 | 预期效果 |
|------|---------|
| 输入裁剪与 Prompt 压缩（RAG 截断、去冗余） | 降低 30–50% 输入 Token |
| Agent 任务分级（轻量任务用小模型） | 价格差可达 10–100×，综合成本大幅下降 |
| 租户级用量埋点（应用层） | 精准成本归因，辅助精细化决策 |
| 批处理合并请求 | 减少请求次数，提升缓存命中率 |

### 5.3 预算规划原则

1. **不囤弹性包**：阿里云百炼弹性包一个月到期清零，按实际需求购买
2. **Agent 上线前压测**：实测真实 Token 用量，乘以 3–10 倍作为安全系数
3. **建立用量监控告警**：在月度配额 70% 时触发告警，避免服务中断

## 六、选型建议

| 企业场景 | 推荐方案 | 核心理由 |
|---------|---------|---------|
| 国内中小团队，用量可预期 | **阿里云百炼 Token Plan** | 人民币结算、Credits 透明、三档坐席灵活、无数据出境问题 |
| 大型企业，需合规/多租户隔离 | **Azure OpenAI PTU 混合** | 独立 Deployment 实现原生租户隔离，微软 SLA 保障 |
| 高并发批量离线任务 | **DeepSeek API + 错峰调度** | 非高峰价格最低，适合夜间批处理场景 |
| 研究/开发，重 Prompt 复用 | **Anthropic API** | 缓存折扣最激进（90%），系统 Prompt 长期复用收益极大 |
| 严禁数据出境 | **私有化部署 / 本地模型** | 任何云 API 均存在数据离境风险 |

**通用红线**

- ❌ 禁用 AI 中转平台
- ❌ 禁止将敏感业务数据发送至未签署数据处理协议的第三方平台
- ❌ 禁止在未压测实际用量前大额购买弹性包
