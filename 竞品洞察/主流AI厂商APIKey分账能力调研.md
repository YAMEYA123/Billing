# 主流 AI 模型厂商基于 API Key 的分账能力调研

> 调研日期：2026-07-15  
> 方法论：多源深度调研 + 对抗性三票验证（105 个子智能体，412 次工具调用）  
> 可信度标注：✅ 高可信（官方文档交叉印证）| ⚠️ 中可信（二级来源）| — 未覆盖（无充分一手资料）

---

## 一、核心结论

主流 AI 厂商的 API Key 分账能力存在明显分化，可归纳为三个层级：

| 层级 | 厂商 | 特征 |
|------|------|------|
| **较完整** | OpenAI、Anthropic | 原生子 Key 体系 + 用量归因 + 配额限流，但不支持下游客户独立出账 |
| **席位制** | 阿里云百炼、百度千帆 | 每席位绑定一个 API Key + 成员维度用量统计，限流合并计算 |
| **工程补偿** | Azure OpenAI | 通过独立 Deployment + API Management 可实现最接近多租户计费，但需大量工程投入 |
| **待明确** | 火山引擎、DeepSeek、腾讯混元、智谱AI、Moonshot | 无充分一手资料支撑，不作结论 |

**共同短板**：所有主流平台均**不支持对下游客户原生独立出账**（即 ISV/SaaS 场景下的二次分账），企业须自建代理层解决。

## 二、各厂商详细能力

### 2.1 OpenAI Projects ✅ 高可信

**数据来源**：OpenAI 官方帮助文档 + Admin API 文档 + Rate Limits API

#### 子账号 / 子 API Key

- 支持 **Projects** 体系，每个 Project 有独立 API Key（含 Service Account 自动生成）
- Project 间资源完全隔离，Key 仅能访问本 Project 资源

#### 用量统计与成本归因

- **Usage API** 和 **Costs API** 均支持按 `project_id` 过滤与分组
- 支持创建支出告警（Spend Alerts），可按项目维度设置阈值并触发邮件通知

#### 配额限制与限流

- Admin API 支持编程式配置每项目、每模型的多维限额：
  - `RPM`（每分钟请求数）
  - `RPD`（每日请求数）  
  - `TPM`（每分钟 Token 数）
  - `TPD`（每日 Token 数）
  - 图像、音频等专项限额
- API 端点：`POST /v1/organization/projects/{project_id}/rate_limits/{rate_limit_id}`

#### 下游客户独立出账

- ❌ **不支持**。计费统一归属于组织层，无法为下游客户原生出账。

#### 能力矩阵

| 能力 | 支持情况 |
|------|---------|
| 子 Key 隔离 | ✅ Project 级 |
| 用量统计 | ✅ Project + 模型维度 |
| 成本归因 | ✅ project_id 过滤 |
| 配额/限流 | ✅ 多维编程式配置 |
| 下游客户独立出账 | ❌ |

---

### 2.2 Anthropic Workspaces ✅ 高可信

**数据来源**：Anthropic 官方 Workspaces 文档 + Usage & Cost API 文档

#### 子账号 / 子 API Key

- 每个组织最多支持 **100 个 Workspace**
- 每个 Workspace 的 API Key 仅能访问本 Workspace 资源

#### 用量统计与成本归因

- Usage & Cost API 支持按 **`workspace_id`** 和 **`api_key_id`** 两个维度过滤与分组
- 官方文档明确说明支持「按 Workspace 进行费用归因（chargeback）」

#### 配额限制与限流

- 每个 Workspace 可独立设置：
  - 月度支出上限（低于组织总上限）
  - RPM 限额
  - 输入 TPM 限额
  - 输出 TPM 限额

#### 下游客户独立出账

- ❌ **不支持**。Workspace 没有独立计费账户，所有费用统一在组织层结算。

#### 能力矩阵

| 能力 | 支持情况 |
|------|---------|
| 子 Key 隔离 | ✅ Workspace 级（最多 100 个） |
| 用量统计 | ✅ workspace_id + api_key_id 双维度 |
| 成本归因 | ✅ 支持 chargeback 归因 |
| 配额/限流 | ✅ 月度支出上限 + RPM + TPM |
| 下游客户独立出账 | ❌ |

---

### 2.3 Azure OpenAI ✅ 高可信

**数据来源**：微软官方多租户架构文档 + Azure API Management 文档

#### 两种模式的能力差异

**独立 Deployment 模式**（推荐企业场景）：
- 为每个租户创建独立 Deployment → 实现 TPM 配额硬隔离
- 按 Deployment 追踪 Token 用量 → 实现精确成本归因
- 可结合 PTU（预留吞吐量）为特定客户预留资源

**共享实例模式**：
- 平台层**不执行**访问控制，必须在应用代码层自行追踪每租户 Token 消耗
- 官方原文：「Azure OpenAI 不在模型部署层执行访问控制，应用必须自行执行租户到部署的访问规则」

#### Azure API Management 增强能力

- `llm-token-limit` 策略：以 `subscription key`（`counter-key=@(context.Subscription.Id)`）为粒度执行 Token 速率限制与配额
- `llm-emit-token-metric`：将用量指标发送至 Azure Monitor，构成多租户计费架构基础

#### 下游客户独立出账

- ❌ 平台层**不原生支持**。完整的独立出账仍需自建账单系统，工程量大。

#### 能力矩阵

| 能力 | 独立 Deployment | 共享实例 |
|------|--------------|---------|
| 子 Key 隔离 | ✅ 每租户独立 Deployment | ⚠️ 应用层实现 |
| 用量统计 | ✅ 按 Deployment | ⚠️ 应用层追踪 |
| 成本归因 | ✅ 精确到租户 | ⚠️ 应用层计算 |
| 配额/限流 | ✅ TPM 硬隔离 | ✅ APIM subscription 粒度 |
| 下游客户独立出账 | ❌（需自建） | ❌（需自建） |

### 2.4 阿里云百炼（Token Plan 团队版）✅ 高可信

**数据来源**：官方帮助文档（token-plan-overview + rate-limit）

#### 子账号 / 子 API Key

- **席位制**：每个席位绑定一名团队成员 + 一个独占 API Key，不可共享
- 管理员统一分配 API Key，成员无法自行生成

#### 用量统计与成本归因

- 用量分析模块提供三维度统计：
  - **成员维度**：每人 Credits 消耗明细
  - **席位维度**：每席位用量趋势
  - **模型维度**：各模型调用量
- 支持 1天 / 7天 / 30天 周期查看

#### 配额限制与限流

- 席位月度配额独立：标准 25,000 / 高级 100,000 / 尊享 250,000 Credits
- ⚠️ **限流按主账号维度合并计算**：「账号下所有 RAM 子账号、业务空间和 API Key 的调用量合并计算」
- **不支持**对子 Key 设置独立限速，高频子 Key 会挤占整体限流额度

#### 下游客户独立出账

- ❌ **不支持**。席位制仅支持团队内部成本归因，不适用 ISV 二次分账场景。

#### 能力矩阵

| 能力 | 支持情况 |
|------|---------|
| 子 Key 隔离 | ✅ 席位级（每人一 Key） |
| 用量统计 | ✅ 成员 + 席位 + 模型三维度 |
| 成本归因 | ✅ 成员维度 Credits 明细 |
| 配额/限流 | ⚠️ 配额独立，限流合并计算 |
| 下游客户独立出账 | ❌ |

---

### 2.5 百度千帆（Token Plan 企业版）✅ 高可信

**数据来源**：百度云官方文档（2026-06-24 上线）

#### 核心机制

- **席位制**：持有席位的用户在「我的订阅」模块获取专属 API Key
- 管理员可在团队管理后台「用量分析」模块查看：
  - 成员维度消耗明细
  - 席位维度消耗明细
  - 模型维度消耗明细

#### 能力矩阵

| 能力 | 支持情况 |
|------|---------|
| 子 Key 隔离 | ✅ 席位级 |
| 用量统计 | ✅ 成员 + 席位 + 模型三维度 |
| 成本归因 | ✅ |
| 配额/限流 | ⚠️ 未获充分文档支撑 |
| 下游客户独立出账 | ❌ |

## 四、企业自建分账系统方案

### 4.1 为何需要自建

所有主流平台均不支持对下游客户原生独立出账。ISV / SaaS 平台若需要为终端客户出账，必须在应用层建立代理层。

### 4.2 LiteLLM 开源代理层方案 ⚠️ 中可信

**数据来源**：LiteLLM 官方文档

LiteLLM 提供四级多租户层次结构：

```
Organization（企业级，仅 Enterprise 版）
  └── Team（团队，开源版可用）
        └── User（用户，开源版可用）
              └── Virtual Key（虚拟 Key，开源版可用）
```

**核心优势**：
- Team Service Account Key 具有 `team_id` 但无 `user_id`
- 成员移除后 Key 不随之删除，可作为**稳定生产凭证**分配给下游客户
- 支持按 Key 维度设置 Token 预算、限流规则
- Organizations（企业级隔离）为付费功能，Teams/Users/Keys 开源版即可使用

### 4.3 自建分账系统架构参考

```
终端客户请求
      ↓
[API 网关层] ← 统一入口，按客户 Key 路由
      ↓
[代理层（LiteLLM / 自研）]
  ├── Token 计量（实时埋点）
  ├── 配额检查（预算上限）
  ├── 限流控制（客户级 RPM/TPM）
  └── 成本归因（写入计费数据库）
      ↓
[上游 AI 厂商 API]
      ↓
[计费系统]
  ├── 按客户出账
  ├── 利润率计算（标记价 vs 成本价）
  └── 账单推送
```

### 4.4 关键工程考量

| 维度 | 要点 |
|------|------|
| **计量精度** | 在响应流结束后异步写入用量，避免同步计量影响延迟 |
| **配额一致性** | 分布式场景下用 Redis 做 Token 桶，避免超配 |
| **模型升级同步** | 上游厂商定价/模型变更时需同步更新计费规则 |
| **成本加价策略** | 建议按模型分类设置加价率（如输入 1.3×、输出 1.5×） |
| **账单对账** | 定期对比自建计量与上游账单，误差 > 5% 时触发告警 |

## 五、选型建议

| 场景 | 推荐方案 |
|------|---------|
| 内部多项目成本管控，需编程式限流 | **OpenAI Projects**（Admin API 最灵活） |
| 内部多团队 chargeback，看重归因粒度 | **Anthropic Workspaces**（api_key_id 维度最细） |
| 大型企业，需真正多租户隔离，预算充足 | **Azure OpenAI 独立 Deployment + APIM** |
| 国内团队，人民币结算，席位制管控 | **阿里云百炼 Token Plan**（注意限流合并） |
| ISV / SaaS，需对下游客户独立出账 | **自建代理层（LiteLLM 或自研）+ 任意上游** |
