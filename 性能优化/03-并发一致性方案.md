# 并发一致性方案

## 1. 并发风险清单

多 Pod 下需要覆盖：

1. 数据库提交成功、Kafka Offset 提交前崩溃；
2. Rebalance 前后旧、新 Pod 重复处理同一段 Offset；
3. 两个 Pod 并发更新同一聚合行；
4. 聚合累计与窗口固化同时发生；
5. 多个定时任务重复固化窗口；
6. 多个 Worker 领取同一话单；
7. Kafka 已发送成功、Outbox 尚未标记成功时崩溃；
8. 同一业务 `messageId` 以不同 Kafka Offset 出现。

## 2. Kafka 分区所有权

所有 Pod 使用同一个 Consumer Group。原始消息以稳定 `billing_key` 分区，正常情况下同一个聚合键只由一个分区、一个 Pod 顺序处理。

Kafka 分区所有权只能降低并发概率，不能代替数据库幂等，因为故障重放和 Rebalance 仍可能重复处理。

## 3. 分区级数据库 Checkpoint

参考结构：

```sql
billing_partition_checkpoint (
    consumer_group,
    topic,
    partition_id,
    next_offset,
    owner_epoch,
    version,
    updated_at
)
```

唯一约束：

```text
(consumer_group, topic, partition_id)
```

处理连续批次 `[start_offset, end_offset)` 时：

| 条件 | 处理 |
|---|---|
| `start == next_offset` | 正常累计并推进 Checkpoint |
| `end <= next_offset` | 整批已经处理，跳过 |
| `start > next_offset` | Offset 缺口，停止并告警 |
| `start < next_offset < end` | 部分重叠，从 Checkpoint 重新读取或裁剪 |

不能只依赖唯一的 `(start_offset, end_offset)` 批次 ID，因为两个不同批次可能部分重叠。

## 4. 原子累计

禁止使用“读取旧值、Go 相加、再写回”。数据库 Upsert 必须直接表达增量：

```text
input_tokens  = current.input_tokens  + delta.input_tokens
output_tokens = current.output_tokens + delta.output_tokens
request_count = current.request_count + delta.request_count
```

原子累计解决并发丢失更新，Checkpoint 解决同一批次重复累计，两者缺一不可。

## 5. Rebalance 协议

### Assign

1. 查询数据库 `next_offset`；
2. Consumer Seek 到该位置；
3. 初始化该分区 Worker；
4. 开始处理。

### Revoke

1. 停止该分区接收新消息；
2. 在有限时间内完成当前数据库事务；
3. 尝试刷新已形成的连续批次；
4. 超时则放弃未提交内存状态；
5. 新 Pod 从数据库 Checkpoint 重放。

回调中不能查询大量元数据、发送话单或执行长时间扫描。

## 6. 窗口固化并发

建议按 `site_id + window_start + virtual_bucket` 管理状态：

```text
OPEN → WAITING → FINALIZING → FINALIZED
                              └→ CORRECTION_PENDING
```

Pod 使用数据库条件更新或租约获得固化权。累计和固化必须遵循相同锁顺序，例如：

```text
窗口桶控制行 → 聚合行 → 分区 Checkpoint
```

进入 `FINALIZING` 后，新的迟到数据不能继续静默修改已生成的话单，应进入修正累计或异常对账。

## 7. Outbox 并发

MySQL 8/PostgreSQL 优先使用 `FOR UPDATE SKIP LOCKED` 短事务领取；不支持时使用 `lease_owner + lease_expired_at` 条件更新。

领取事务提交后再执行元数据查询和 Kafka 发送，不能持有数据库锁进行网络调用。

## 8. Redis 去重边界

- 相同 Offset 重放：由 DB Checkpoint 处理。
- 相同 `messageId` 出现在不同 Offset：由生产端稳定 ID、Kafka 幂等 Producer 和短期 Redis 去重共同处理。

Redis 必须使用原子 `SET NX EX` 和 Pipeline。但需要特别审计“Redis 标记成功、数据库失败”的少计窗口；未解决前不能直接删除 Redis 去重。

## 9. 正确性验收

- 相同 Offset 重放后 Token 不增加；
- 两个 Pod 重叠处理时最终值正确；
- 数据库提交后强杀，恢复后不重复累计；
- 数据库提交前强杀，恢复后不漏计；
- 窗口固化与累计并发时没有少计；
- 相同 `bill_id` 重发时下游仅结算一次。
