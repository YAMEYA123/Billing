# 数据库与分区方案

## 1. 表职责

建议拆分为：

| 表 | 职责 |
|---|---|
| `usage_5m_aggregate` | 权威五分钟聚合事实 |
| `billing_partition_checkpoint` | Kafka 分区已入库进度 |
| `billing_window_control` | 窗口桶生命周期和固化租约 |
| `billing_outbox` | 待发送、重试中的话单任务 |
| `billing_report_history` | 已上报话单审计历史 |

## 2. 聚合事实表

参考字段：

```text
window_start, virtual_bucket, site_id, user_id, model_id, api_key_id,
input_tokens, output_tokens, request_count, revision,
created_at, updated_at
```

业务唯一键：

```text
(window_start, virtual_bucket, site_id, user_id, model_id, api_key_id)
```

金额应使用最小货币单位或定点数，Token 和次数使用整数，禁止浮点累计。

## 3. 日分区

业务窗口保持 5 分钟，数据库物理分区使用 1 天：

```text
usage_5m_aggregate
├── 2026-07-22
├── 2026-07-23
└── 2026-07-24
```

不建议每五分钟一个物理分区，也不建议应用动态拼日期表名。每天 288 个五分钟窗口共用一个日分区。

收益：

- 最近窗口查询可分区裁剪；
- 过期数据可按分区归档；
- 避免超大范围 `DELETE`；
- 单个索引和维护任务规模受控。

限制：时间分区不能提高单实例总写能力，不能替代未来物理分库。

## 4. MySQL/PostgreSQL 差异

- PostgreSQL 使用 `ON CONFLICT DO UPDATE`；
- MySQL 使用 `ON DUPLICATE KEY UPDATE`；
- 两者的分区表唯一约束都需结合具体版本验证；
- PostgreSQL 可使用待处理状态的部分索引；
- MySQL Outbox 通常使用 `(status, next_retry_at, id)` 复合索引；
- `SKIP LOCKED` 能力需要根据站点数据库版本确认。

应通过统一 Repository 契约测试保证两种实现语义一致，而不是追求 SQL 字面一致。

## 5. Outbox 索引与生命周期

查询条件：

```text
status IN (PENDING, RETRY)
AND next_retry_at <= now
```

已发送记录只在热表保留有限时间，然后迁入 `billing_report_history`。归档初期只迁移不删除，完成恢复演练后再启用物理清理。

## 6. 分区维护

- 提前创建未来 7～14 天分区；
- 对缺失分区、默认分区写入和异常增长告警；
- 记录每个分区的行数、表大小、索引大小；
- 归档后验证行数、Token 总量和校验结果；
- 聚合和话单历史分别制定符合法务/财务要求的保留期。

## 7. 虚拟桶与分库

现在预留：

```text
virtual_bucket = stable_hash(billing_key) % 256
```

当前所有桶仍写一个数据库。出现以下条件时才启动物理分库：

- 单库持续达到压测安全容量 60%～70%；
- Redo/WAL、IOPS 或锁等待已成为稳定瓶颈；
- 日分区、批量写、归档后仍不能满足目标；
- 一个扩容周期内预计突破安全线。

迁移按少量桶逐批执行：回填、双写、校验、切读、切写、观察、清理。路由配置必须版本化并支持切回。

## 8. 风险与回滚

- 新分区表采用双写、回填、逐窗口 Token 对账后切换；
- 不原地重构生产大表；
- 新增表和索引优先，旧表在完整观察期内保留；
- 分区删除属于破坏性操作，必须先归档并演练恢复。
