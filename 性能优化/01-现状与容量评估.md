# 现状与容量评估

## 1. 已知参数

| 维度 | 当前/目标 |
|---|---|
| 服务语言 | Go |
| 当前调用量 | 约 5,000 QPS |
| 目标调用量 | 约 50,000 QPS |
| Kafka 分区 | 4 |
| Consumer 实例 | 4 个 Pod |
| 聚合窗口 | 固定自然时钟 5 分钟 |
| 当前窗口唯一键 | 约 800 |
| 规划窗口唯一键 | 1 万～10 万 |
| 数据库 | 不同站点分别使用 MySQL、PostgreSQL |
| 原始消息保留 | 3 天 |
| 上报 SLO | 窗口结束后 5 分钟内 |

## 2. 容量模型

50,000 QPS 下，每个 5 分钟窗口原始消息上界为：

```text
50,000 × 300 = 15,000,000 条
```

每天固定有：

```text
24 × 60 ÷ 5 = 288 个窗口
```

最终聚合行数由窗口唯一键数 `K` 决定：

| K | 每日新增聚合行 | 每月约新增行 | 平均话单输出 |
|---:|---:|---:|---:|
| 10,000 | 288 万 | 8,640 万 | 33 条/秒 |
| 50,000 | 1,440 万 | 4.32 亿 | 167 条/秒 |
| 100,000 | 2,880 万 | 8.64 亿 | 333 条/秒 |

数据库实际写压力不仅由最终行数决定，还由刷新频率决定。若每 5 秒刷新一次，同一活跃行可能在一个窗口内被更新约 60 次。因此要同时监控：

- 原始消息数；
- 窗口唯一键数；
- 聚合比 `消息数 / 聚合行数`；
- 每窗口实际 Upsert 次数；
- 单批事务行数及耗时。

## 3. 可能的第一瓶颈

优先排查顺序：

1. Redis 逐事件去重产生的网络和内存压力；
2. 消费线程被数据库或元数据调用阻塞；
3. 同一窗口行被过于频繁地 Upsert；
4. Kafka 单分区消费能力不足；
5. 话单窗口关闭时的元数据查询尖峰；
6. 聚合历史表索引和维护成本；
7. 热点客户导致分区或数据库热点。

## 4. 必须补采的基线

### Kafka

- 每分区生产、消费速率；
- Consumer Lag、Lag 增长率和追平时间；
- Rebalance 次数、耗时、移动分区数；
- 单消息平均大小；
- 最热 Key 和最热分区占比。

### Go 服务

- Poll、反序列化、聚合、刷新各阶段耗时；
- 每批消息数和聚合后键数；
- Heap、GC、CPU、锁竞争；
- Redis 请求量、Pipeline 大小、命中率；
- 数据库连接池等待。

### 数据库

- Upsert P95/P99；
- 每秒事务和实际更新行数；
- 行锁、死锁、WAL/Redo、IOPS；
- 表和索引增长；
- 按窗口查询扫描行数。

## 5. 容量决策规则

Kafka 分区数不按 QPS 倍数机械增长，而按压测确定：

```text
所需分区数 = 峰值QPS ÷ 单分区安全QPS × 安全系数
```

数据库分库也不按总行数立即触发，建议在以下条件同时出现时启动：

- 稳态资源达到压测安全容量的 60%～70%；
- 一个扩容周期内预计突破安全线；
- 日分区、批量写和归档后仍无法满足写入或恢复目标。
